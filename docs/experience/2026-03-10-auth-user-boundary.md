# Auth 与 User 职责边界沉淀

**日期**：2026-03-10  
**类型**：Experience  
**适用范围**：`apps/ww-server/src/auth/*`、`apps/ww-server/src/user/*`

---

## 背景

`auth + user` 这一条线已经不是空壳。
它已经承担了注册、登录、资料查询、资料更新和 JWT 认证。

这类代码在项目早期通常能一起跑起来，但跑通不等于边界稳定。
当职责继续长在同一个 `service` 里时，后面每加一个需求，判断成本都会变高。

---

## 本轮看到的主要信号

在这轮处理前，最明显的几个信号是：

- `UserService` 同时承担注册、登录、用户资料和消费记录查询
- token 签发逻辑留在 `UserService` 内部，认证语义和用户领域语义混在一起
- `JwtStrategy.validate` 返回结构可用，但缺少更明确的载荷类型边界
- `user.controller` 的登录入口虽然能工作，但控制器委托方向还不够清楚

这些信号说明，当前最先该收的不是“功能”，而是“责任归属”。

---

## 本轮怎么处理

### 1. 把登录与 token 签发收回 `AuthService`

这轮没有做大拆分，也没有强行新建完整 `AuthModule` 体系。
我们先做轻量收束：

- `AuthService` 负责登录流程编排
- `AuthService` 负责签发 JWT
- `UserController.login()` 直接委托给 `AuthService`

这样做以后，登录成功返回的 `token + user` 就更符合认证语义。

### 2. 让 `UserService` 回到用户领域职责

本轮之后，`UserService` 主要保留：

- 注册
- 校验用户凭证
- 用户对象脱敏
- 获取和更新用户资料
- 查询消费记录

这里有一个重要判断：

> “校验凭证”可以暂时留在 `UserService`，因为它仍然贴着用户领域对象；
> “签发 token” 应该回到 `AuthService`，因为它属于认证结果。

### 3. 给 JWT 载荷补上明确类型

本轮新增了：

- `JwtPayload`
- `AuthenticatedUser`

这一步不大，但很关键。
它让 `JwtStrategy.validate()` 不再只是“返回一个对象”，而是在表达：

- token 里放了什么
- 请求上下文里应该拿到什么

类型边界清楚以后，后面再扩展角色、权限或审计信息会更稳。

---

## 为什么这轮不直接大拆 `UserService`

这轮故意没有继续拆消费记录、资料查询或其它用户相关职责。
原因不是这些问题不存在，而是当前还没到“一步拆干净”的时候。

现在最重要的是先立住第一条硬边界：

> 认证负责认证，用户负责用户。

只要这条边界立住，下一轮再看 `UserService` 是否继续超载，判断就会更准。

如果在这一步同时做登录拆分、consumption 拆分、资料查询拆分，很容易把一轮“边界澄清”做成一轮“大型整理”。

---

## 经验结论

### 经验一：认证成功结果和用户领域对象，不是一回事

登录返回里同时出现 `token` 和 `user` 很常见。
但这不代表它们应该由同一个服务负责生成。

- `user` 是用户领域对象的对外表示
- `token` 是认证层给出的通行结果

两者可以一起返回，但不应该混成一个职责。

### 经验二：轻量收边界，比一次大拆更适合当前阶段

当前项目还在快速生长期。
这时候最有价值的动作，不是一次性把模块切得非常“完整”，而是先把最容易混淆的责任拆清楚。

这轮选择轻量引入 `AuthService`，就是为了先把最关键的边界拉直。

### 经验三：类型声明本身也是边界的一部分

`JwtPayload` 和 `AuthenticatedUser` 不是为了“看起来更规范”。
它们是在告诉后来的人：

- token 只承载这些字段
- 守卫通过后，请求上下文只保证这些字段

边界一旦写进类型，后续误用就会少很多。
