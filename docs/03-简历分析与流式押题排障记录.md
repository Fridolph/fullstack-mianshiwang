# 03 - 简历分析与流式押题排障记录

## 文档目的

这份文档记录本次 `apps/web-nuxt` 与 `apps/ww-server` 联调时遇到的关键故障、定位过程、修复方式和验证结果。

它重点回答 4 个问题：

- 为什么前端登录后，简历分析和流式押题会失败
- 为什么浏览器里会出现 `stream has been aborted`
- `requestId` 应该由前端生成，还是由后端补写
- 本次联调后，哪些链路已经恢复正常

---

## 一、问题现象

本次联调主要出现了 3 类问题。

### 1. 简历分析接口前端拿不到正确数据

前端调用 `POST /api/interview/analyze-resume` 后，页面虽然能收到响应，但业务层拿到的数据结构和页面预期不一致，导致 `sessionId` 与 `analysis` 取值异常。

### 2. 简历押题流式接口在前端表现为中断

前端调用 `POST /api/interview/resume/quiz/stream` 时，浏览器网络面板会看到请求建立成功，但页面会提示 `stream has been aborted`，结果页也无法稳定跳转。

### 3. 手工调试时，接口偶发 400 或保存失败

手工用 `curl` 调试流式接口时，如果没有传 `requestId`，后端会因为 DTO 校验失败直接返回 `requestId must be a UUID`。

即使补上 `requestId`，后端在接入真实 AI 生成后，也会因为题目分类和难度字段不符合 Mongo schema 的枚举要求，导致结果保存失败。

---

## 二、根因拆解

### 1. `analyze-resume` 和 `continue-conversation` 返回结构多包了一层

后端已经有统一响应结构 `{ code, message, data }`，前端请求插件也会自动拆掉最外层。

但是 `apps/ww-server/src/interview/interview.controller.ts` 之前在业务层又额外包了一层 `data`，导致前端实际拿到的是“二次嵌套”的对象。

### 2. SSE 成功路径没有发出 `complete`

`apps/ww-server/src/interview/services/interview.service.ts` 之前在成功路径里没有稳定发出 `complete` 事件，前端读流结束时只能看到连接被关闭，无法把这次流程识别成“成功结束”。

这就是浏览器里 `stream has been aborted` 的核心诱因之一。连接本身并不一定是异常断开，而是前端没有拿到一个明确的完成信号。

### 3. 前端旧连接状态清理不完整

`apps/web-nuxt/app/pages/interview/start.vue` 在重新发起流式请求时，会关闭旧连接，但旧的运行态没有在所有完成/失败分支里及时清空。

这会放大“流已经结束，但页面仍以为有活动连接”的假象。

### 4. 流式押题此前仍然使用空的占位 AI 结果

`apps/ww-server/src/interview/services/interview.service.ts` 原来在真正保存结果前，仍然使用 `const aiResult: any = {}` 这种占位写法。

这意味着即使流式接口走通，最终保存的题目列表也是空数组，前端结果页自然无法展示真实题目。

### 5. 真实 AI 输出和 Mongo schema 枚举不一致

在接入真实 AI 生成后，我复现到新的保存错误：

- AI 返回的 `category` 是 `💻 技术能力`、`🎯 项目经验`、`🧠 问题解决`
- AI 返回的 `difficulty` 是 `Easy`、`Medium`、`Hard`

但 `apps/ww-server/src/interview/schemas/interview-quiz-result.schema.ts` 期望的是：

- `category`: `technical`、`project`、`problem-solving` 等英文枚举
- `difficulty`: `easy`、`medium`、`hard`

因此 Mongoose 在保存 `ResumeQuizResult` 时直接校验失败。

### 6. `requestId` 的责任方是“发起请求的一侧”

`requestId` 不是后端在收到请求后再补写的字段，而是幂等键。

幂等键必须在请求进入服务端之前就已经存在，这样后端才能判断“这是新请求”还是“同一请求的重复提交”。

因此：

- 浏览器前端发请求时，前端生成 `requestId`
- Postman 或 `curl` 手工调试时，调用方自己生成 `requestId`
- 后端只负责校验、消费和基于它做幂等判断

当前前端已经在 `apps/web-nuxt/app/pages/interview/start.vue` 中通过 `crypto.randomUUID()` 自动生成了 `requestId`。如果是手工调试，需要自己显式传入。

## 三、本次修复内容

### 1. 修正简历分析和继续追问的响应结构

我在 `apps/ww-server/src/interview/interview.controller.ts` 中统一改为：

- `analyze-resume` 直接返回 `ResponseUtil.success(result, ...)`
- `continue-conversation` 直接返回 `ResponseUtil.success({ response: result }, ...)`

这样前端请求插件拆包后，页面能直接拿到单层的业务数据。

### 2. 补齐 SSE 的完成信号

我在 `apps/ww-server/src/interview/services/interview.service.ts` 中统一整理了完成事件发送逻辑：

- 缓存命中时也发送 `complete`
- 正常生成成功时发送 `complete`
- 出错时发送 `error`

前端现在可以明确区分“成功结束”和“异常中断”。

### 3. 调整前端流状态清理

我在 `apps/web-nuxt/app/pages/interview/start.vue` 中补齐了连接清理逻辑：

- `complete` 时清空 `activeConnection`
- `error` 时清空 `activeConnection`
- `onError` / `onComplete` 回调里也同步清理

这样页面不会再残留一条已经失效的旧连接状态。

### 4. 接入真实 AI 生成链路

我在 `apps/ww-server/src/interview/services/interview.service.ts` 中接入了真实的题目生成与匹配分析流程：

- 使用 `PromptTemplate`
- 使用 `StructuredOutputParser`
- 使用 `AIModelFactory`
- 分别生成题目结果和匹配分析
- 合并后保存到 `ResumeQuizResult`

这一步替换掉了原来的空占位 `aiResult`。

### 5. 增加 AI 输出归一化，解决枚举校验失败

我在 `apps/ww-server/src/interview/services/interview.service.ts` 中新增了分类和难度规范化逻辑，把 AI 常见输出映射为 schema 可接受的枚举值。

例如：

- `💻 技术能力` -> `technical`
- `🎯 项目经验` -> `project`
- `🧠 问题解决` -> `problem-solving`
- `Easy` -> `easy`
- `Medium` -> `medium`
- `Hard` -> `hard`

这样即使模型偶尔输出中文标签或带 emoji，也不会再因为枚举不一致而保存失败。

### 6. 补强 Prompt 约束

我在 `apps/ww-server/src/interview/prompts/resume-quiz.propmts.ts` 中补充了两类约束：

- 题目生成 Prompt 明确要求 `category` / `difficulty` 输出固定英文枚举
- 匹配分析 Prompt 补齐了缺失的角色说明、输入上下文和结构化输出要求

这可以减少模型偏离约定格式的概率。

### 7. 补齐保存字段和错误提示

我还补了两处容易遗漏的细节：

- 保存 `ResumeQuizResult` 时补上 `missingSkills`
- `resumeURL` 解析路径返回清理后的文本，并把最终报错改成“请提供简历内容或可访问的简历 URL”

## 四、与 `requestId` 相关的结论

这里单独再强调一次，方便后续联调时统一口径。

### 正确做法

由请求发起方生成 `requestId`，再随请求一起发给后端。

### 为什么不能让后端事后补写

如果后端是在收到请求以后才生成 `requestId`，那它在真正处理请求前就无法知道：

- 这是第一次点击
- 还是用户重复点击
- 还是浏览器重试
- 还是网络抖动造成的重复提交

也就是说，幂等判断会失效。

### 当前项目里的实际落点

- 浏览器前端：`apps/web-nuxt/app/pages/interview/start.vue`
- 手工调试：`curl` / Postman 调用者自己传
- 后端消费记录：`apps/ww-server/src/interview/services/interview.service.ts`

## 八、Flashpost / Postman 手动调试说明

手动调试 `POST /api/interview/resume/quiz/stream` 时，最容易踩的坑不是请求体，而是 `requestId`、鉴权头和 SSE 的使用方式。

### 1. 必须带上的请求头

这条接口至少要带上下面两个头：

- `Authorization: Bearer <token>`
- `Accept: text/event-stream`

请求体仍然使用 JSON，所以你也应该保留：

- `Content-Type: application/json`

### 2. 每次手动发送都要换一个新的 `requestId`

后端把 `requestId` 作为幂等键使用。同一个用户如果重复提交同一个 `requestId`，后端会先检查这条记录的状态：

- `pending`：返回“请求正在处理中，请稍后查询结果”
- `success`：直接命中缓存结果
- `failed`：说明上次处理失败

这意味着手工调试时，你不能把同一个 body 连续发送多次，除非你就是想复用那次请求的结果。

### 3. Flashpost / Postman 的推荐流程

你可以按下面顺序调试：

1. 先登录，拿到 JWT token。
2. 构造新的 `requestId`，每次发送前都换一个新的 UUID。
3. 调 `POST /api/interview/resume/quiz/stream`，观察 SSE 返回的 `progress` / `complete` / `error`。
4. 如果中途断开，或者你怀疑还在处理中，再调用状态查询接口确认当前状态。

### 4. 新增的状态查询接口

为了方便手工联调，后端新增了一个查询接口：

- `GET /api/interview/resume/quiz/status?requestId=<uuid>`

这条接口同样需要登录态。它会返回当前 `requestId` 对应的处理状态：

- `pending`
- `success`
- `failed`

当状态是 `success` 时，响应里会直接附带对应的结果明细，包括：

- `resultId`
- `summary`
- `questions`
- `matchScore`
- `matchedSkills`
- `missingSkills`
- `learningPriorities`
- `radarData`

### 5. 一个可直接参考的调试方式

当你在 Flashpost 里手动发流式请求时，可以这样理解：

- `stream` 接口负责“发起生成”
- `status` 接口负责“查询这一单现在走到哪一步”

这种组合特别适合手工联调，因为你不需要每次都盯着一条长连接看是否结束。
