# `Task 4` 收口：第二个真实 Dao 样本已经站住了

---

## 这一步意味着什么

`Task 4` 做完，说明 `Dao` 工程体系在 `fullstack-mianshiwang` 里不再只有第一条样本。
它已经在另一条完全不同的业务线上，再次跑出一条完整闭环。

这件事很重要。
因为第一条样本可以被解释为“刚好选对了地方”，第二条样本跑通以后，才更接近“这套方法开始具备可迁移性”。

---

## 这次闭环落在什么地方

`Task 4` 选的是 `auth + user`。
这条线和 `Task 3` 很不一样。

`Task 3` 主要面对的是：

- 业务入口边界
- AI 调用边界

而 `Task 4` 面对的是：

- 认证职责边界
- 用户域职责边界
- 查询职责是否开始超载

也就是说，第二个样本不是重复第一条路，而是在验证这套判断语言能不能迁移到更基础的业务骨架上。

---

## 这次我们做了哪两刀

### 第一刀：先收认证边界

这一步把登录流程和 token 签发从 `UserService` 中收回 `AuthService`。
同时为 JWT 载荷和认证后用户补上了更明确的类型边界。

对应收口：

```text
[☶☵][蹇] refactor(auth): 收紧 token 签发与用户职责边界
```

### 第二刀：再收查询职责边界

这一步把消费记录查询和聚合统计从 `UserService` 中拿出来，单独收进 `UserConsumptionQueryService`。
这样 `UserService` 回到更纯的用户领域职责，查询线也有了更自然的落点。

对应收口：

```text
[☱☷][萃] refactor(user): 收束 consumption 查询职责，避免 user service 超载
```

这两刀放在一起，才构成 `auth + user` 这条第二样本闭环。

---

## 为什么这条样本也成立

回头看，`Task 4` 最关键的不是“拆了两个 service”。
而是判断顺序仍然是对的。

我们没有一上来就把 `user` 线拆成很多模块。
而是：

- 先收最核心的认证边界
- 再看剩下的查询职责是不是开始偏离主语义
- 每一步都独立验证
- 每一步都同步补经验和博客

这种顺序，和 `Task 3` 的内核是一致的。
也正因为顺序一致，第二条样本才能真正说明：

> `Dao Coding -> 实现 -> 验证 -> 沉淀 -> Dao Commit` 这条链路不是偶然奏效。

---

## `Task 3` 和 `Task 4` 放在一起，已经能看见什么

到这里，仓库里已经有两条不同风格的真实样本：

### 样本一：`interview + ai`

重点验证：

- 入口边界
- AI facade 边界

### 样本二：`auth + user`

重点验证：

- 认证边界
- 用户域主职责
- 查询职责收束

这两条样本放在一起以后，已经可以看出一个稳定模式：

1. 不先追求大而全抽象
2. 先找最先失语的边界
3. 先切最值钱的一刀
4. 每一刀都独立收口成经验与提交

这就是目前这个仓库里，最真实的 `Dao` 工程节奏。
